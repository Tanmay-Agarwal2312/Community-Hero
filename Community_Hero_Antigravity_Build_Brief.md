## 1. Project Overview

Community Hero is a civic issue reporting platform. Citizens report local problems (potholes, garbage, broken streetlights, water leaks, road damage) with a photo and location. An AI vision model auto-categorizes the issue. Other citizens verify reports are real via upvotes/downvotes. Once an authority (or, for certain categories, the original community) marks an issue fixed, the community confirms the fix before it's marked fully resolved. A public map and dashboard make the whole pipeline transparent.

Build this as a MERN application (React frontend, Node.js/Express backend, MongoDB database). This is a 24-hour solo hackathon build — prioritize a working end-to-end flow over exhaustive edge-case handling.

### 1.1 Core feature list

1. **Authentication** — Google Sign-In only, role-based access (citizen / org-admin)
2. **Issue reporting** — photo upload, geolocation capture, AI auto-categorization
3. **Resolution pipeline** — 5-stage state machine with role-gated transitions
4. **Map view** — Leaflet + OpenStreetMap, all issues pinned and color/icon-coded by status
5. **Community verification** — net-vote system gating two pipeline transitions
6. **Duplicate detection** — geospatial + category match, routes into the existing vote system
7. **Impact dashboard** — aggregate stats and charts, backed by seed data
8. **Issue feed/list** — filterable, scrollable list of all issues

## 3. Corrections — What's Wrong in the Reference Images

The reference images are a style guide, not a literal spec. Several details in them are incorrect, inconsistent, or mismatched against this project's actual requirements. **Do not copy these into the build.** Each item below names the image, what's wrong, and what to build instead.

### 3.1 Status label mismatch (Image 6 — Dashboard)

The dashboard's Status Breakdown chart shows four states: Resolved, In Progress, Pending Verification, and **Rejected**. "Rejected" does not exist anywhere in this project's actual pipeline — there is no rejection mechanic defined. The chart is also missing "Verified" entirely, which IS a real stage.

- ❌ Dashboard pie chart shows "Rejected" as a status (2%) → **✅ Remove "Rejected." It is not a real pipeline state for this project — do not invent a rejection mechanic.**
- ❌ Dashboard pie chart is missing "Verified" as a status → **✅ Add "Verified" as one of the chart segments. The chart must show all 5 real pipeline states: Reported, Verified, In Progress, Pending Verification (= Resolved-Pending-Verification), Resolved.**

### 3.2 Ambiguous "Pending Verification" label (Image 3 — Map legend)

The map legend's "Pending Verification" pin could be misread as either of two different moments in the pipeline: (a) a freshly Reported issue awaiting its first 5 net upvotes to become Verified, or (b) a Resolved-Pending-Verification issue where a fix has been claimed and is awaiting confirmation votes. These are different stages with different vote thresholds (5 vs 3) and must not be visually merged.

- ❌ Map legend shows one ambiguous "Pending Verification" pin → **✅ On the map specifically, label this pin "Resolved – Pending Verification" to make clear it refers ONLY to stage (b) above — an issue with a fix claimed, awaiting confirmation. The plain "Reported" pin already represents stage (a).**

### 3.3 Wrong / incomplete category set (Image 2 — Report flow, Image 5 — Feed)

The report flow and feed screens use a 4-category set: Roads, Waste, Lighting, Water. This project's actual category taxonomy (driven by the AI categorization step, see Section 5) has 6 categories.

- ❌ Only 4 categories shown: Roads, Waste, Lighting, Water → **✅ Use the real 6-category set everywhere: `pothole`, `garbage`, `streetlight`, `water_leak`, `road_damage`, `other`. Update category selector UI, filter pills, and any category icon/color mapping to reflect all 6, not 4.**

### 3.4 Incomplete state coverage in mockup screens (Images 4 & 5)

Image 4 (Issue Detail) only shows a timeline reaching "In Progress" — it never shows the later stages. Image 5 (Feed) only shows example badges for "In Progress," "Verified," and "Reported" — it never shows "Pending Verification," "Resolved," or "Disputed." These are just incomplete example data in the mockup, not a deliberate omission.

- ❌ Mockups only demonstrate 3–4 of the 5 real states → **✅ Build the timeline component and feed badge component to support ALL 5 pipeline states plus the separate Disputed overlay flag, even though the static mockups only show a subset.**

### 3.5 Vote UI has no accessible backup for color (Image 4)

The upvote/downvote counter on the issue detail page uses a plain green up-arrow and gray/red down-arrow with no shape or icon differentiation beyond color and direction. This project requires status/voting indicators to be distinguishable without relying on color alone (colorblind accessibility).

- ❌ Vote arrows differentiated by color only → **✅ Keep the up/down arrow shapes (they're already directionally distinct, which is fine) but ensure no other status indicator in the app relies on color alone. Image 3's map legend already does this correctly with distinct icon shapes per status (checkmark, exclamation, dots, magnifying glass, X) — carry that same shape-based approach to every other status indicator in the app, not just the map.**

## 4. Authentication & Roles

### 4.1 Requirements

- Google Sign-In only. No email/password signup, no separate registration form.
- No standalone profile page. A logged-in user is represented by a `users` collection document populated directly from the Google OAuth payload: name, email, photo URL, googleId.
- Add a `role` field on the user document: `"citizen"` (default for all new sign-ins) or `"org_admin"`.
- Org/admin accounts are NOT self-service — there is no admin signup flow. Hardcode 1–2 specific Google account emails as `org_admin` directly in the seed/setup script or via a one-time manual DB update. Do not build an admin registration UI.
- All API routes that mutate issue status must check `req.user.role` server-side, not just hide buttons client-side.

### 4.2 `users` collection schema

| Field | Type | Notes |
|---|---|---|
| `googleId` | String | From OAuth payload, unique |
| `name` | String | From OAuth payload |
| `email` | String | From OAuth payload |
| `photoUrl` | String | From OAuth payload |
| `role` | String enum | `"citizen"` \| `"org_admin"`, default `"citizen"` |
| `createdAt` | Date | Set on first sign-in |

## 5. Issue Reporting & AI Categorization

### 5.1 Report flow

1. Citizen taps "Report an Issue." Upload a photo (required) and an optional short text description.
2. On submission, capture the citizen's current coordinates via the browser Geolocation API (`navigator.geolocation.getCurrentPosition`) — store as `{ lat, lng }`.
3. Send the photo to the Gemini API for categorization (see 5.2 below).
4. Run duplicate detection using the returned category + coordinates (see Section 7) BEFORE final submission — if matches are found, show them to the citizen first.
5. If no duplicates, or citizen confirms "different issue," create the new issue document with status `"reported"`.

### 5.2 AI categorization — Gemini API

- Model: `gemini-2.5-flash-lite` (free tier, no billing/credit card required).
- Single API call per photo upload. Send the image plus a system prompt instructing the model to act as a civic-issue classifier.
- Required structured JSON output:
  - `category`: enum — one of `"pothole"`, `"garbage"`, `"streetlight"`, `"water_leak"`, `"road_damage"`, `"other"`
  - `severity`: enum — `"low"`, `"medium"`, `"high"`
  - `citizen_fixable`: boolean — true for categories a citizen could plausibly fix themselves (e.g. garbage/debris), false for infrastructure issues (pothole, streetlight, water_leak, road_damage) that require a municipal authority
  - `short_description`: a 1-sentence auto-generated caption of the photo
- Prompt Gemini explicitly to return ONLY JSON matching this schema, no preamble or markdown fencing, so the response can be parsed directly.
- Store all 4 returned fields on the issue document at creation time.

### 5.3 `issues` collection schema (core fields)

| Field | Type | Notes |
|---|---|---|
| `reporterId` | ObjectId | ref `users` |
| `photoUrl` | String | Uploaded photo |
| `description` | String | Citizen-entered, optional |
| `location` | `{ lat, lng }` | GeoJSON Point recommended for 2dsphere index |
| `category` | String enum | From Gemini: `pothole` \| `garbage` \| `streetlight` \| `water_leak` \| `road_damage` \| `other` |
| `severity` | String enum | From Gemini: `low` \| `medium` \| `high` |
| `citizenFixable` | Boolean | From Gemini |
| `aiDescription` | String | From Gemini, auto-caption |
| `status` | String enum | `reported` \| `verified` \| `in_progress` \| `resolved_pending_verification` \| `resolved` |
| `disputed` | Boolean | Default `false`. See Section 7 / Section 6.3. |
| `resolutionProofUrl` | String | Set when status moves to `resolved_pending_verification` |
| `resolvedBy` | ObjectId | ref `users` — who submitted the resolution proof (org or citizen) |
| `createdAt` / `updatedAt` | Date | Timestamps for dashboard stats and pipeline display |

## 6. Resolution Pipeline (5-State Machine)

**Reported → Verified → In Progress → Resolved-Pending-Verification → Resolved**

### 6.1 Stage transitions and who can trigger them

| Transition | Triggered by | Condition |
|---|---|---|
| `reported` → `verified` | Community voting (automatic) | 5 net upvotes (upvotes − downvotes) reached |
| `verified` → `in_progress` | `org_admin` only | Org acknowledges and begins work — manual action, button only visible/usable to `org_admin` role |
| `in_progress` → `resolved_pending_verification` | `org_admin` (any category) OR citizen (ONLY if `issue.citizenFixable === true`) | Resolution photo proof submitted |
| `resolved_pending_verification` → `resolved` | Community voting (automatic) | 3 net upvotes reached. Any downvote at this stage also sets `disputed: true` (see 6.3) |

### 6.2 Critical rule — who can submit resolution proof

> **Enforce server-side, not just in the UI.** Citizens CANNOT mark infrastructure issues (pothole, streetlight, water_leak, road_damage) resolved themselves — only an `org_admin` account can submit resolution proof for these. Only when `issue.citizenFixable === true` (set by the Gemini categorization step, typically for garbage/debris-type issues) can any citizen submit resolution proof. This must be checked in the backend route handler, not only by hiding the button on the frontend.

### 6.3 Community verification mechanics

- One vote per account per issue. Maintain a separate `votes` collection: `{ userId, issueId, value: 1 | -1, createdAt }`. Before accepting a new vote, query for an existing vote by the same `userId` + `issueId`; if one exists, either reject the new vote or treat it as an update/retraction — never allow two active votes from the same user on the same issue.
- Net vote count = sum of all vote values for that issue (upvotes count as +1, downvotes as −1).
- `reported` → `verified` requires net votes ≥ 5.
- `resolved_pending_verification` → `resolved` requires net votes ≥ 3 (recalculate net votes freshly for THIS stage — do not carry over the vote count from the earlier `verified` stage; votes at this stage are about confirming the FIX, not confirming the original report).
- Any single downvote received while an issue is in `resolved_pending_verification` ALSO sets `disputed: true` on the issue, regardless of the net vote total. This is a one-way flag for this stage — once true, display the Disputed badge permanently for that resolution cycle (do not auto-clear it even if later votes bring the net positive).
- There is no proximity/location weighting on votes — any logged-in citizen (who hasn't already voted and isn't the original reporter, where relevant) can vote from anywhere. This was a deliberate scope cut, do not add location-based vote weighting.

### 6.4 Seed data requirement

> **This affects every screen, not just the dashboard.** Before final demo, run a seed script that inserts 30–50 fake issues into the `issues` collection with: varied categories (all 6, not clustered on one or two), varied statuses spread across ALL 5 pipeline stages (not just `reported` — make sure some issues exist at `verified`, `in_progress`, `resolved_pending_verification`, and `resolved` so the pipeline visibly works), varied `createdAt` timestamps spread over the past few weeks, and varied `{ lat, lng }` scattered across a real city's coordinate range (not clustered in one spot). Also seed a matching set of `votes` documents so the net-vote counts on seeded issues are internally consistent with their seeded status (e.g. a `verified` issue should have ≥ 5 net votes in the `votes` collection, not zero). Without this, the map looks empty, the dashboard charts look fake/flat, and the pipeline looks unused.

## 7. Duplicate Detection

Goal: prevent the same real-world issue from being reported many times and overwhelming the resolving organization, by routing a citizen who encounters an existing issue into the existing verification system instead of creating a new issue document.

### 7.1 Flow

1. On report submission, after the Gemini categorization step returns a category, query existing issues where `status` is NOT `"resolved"` AND `category` matches AND `location` is within ~50 meters of the new report's coordinates.
2. Implement the proximity check with MongoDB's geospatial query (`$near` or `$geoWithin` with a `2dsphere` index on the `location` field), or a manual haversine distance calculation in the backend if you want to skip the index setup. Test this with two known points ~50m apart before trusting it — verify the radius logic actually behaves as expected.
3. If matches are found, show them to the citizen as "Possible duplicates" before allowing final submission, with two choices: "Yes, same issue" or "No, different issue."
4. **"Yes, same issue"** → do NOT create a new issue document. Instead, add a normal verification upvote (in the existing `votes` collection, `value: 1`) on the existing issue on behalf of the citizen viewing this screen — exactly as if they had upvoted from the issue detail page. This is the SAME vote mechanism described in Section 6.3, not a separate system.
5. Before adding that vote, check: (a) is this citizen the original reporter of the existing issue, or (b) has this citizen already voted on it? If either is true, do not add a vote — show "Already reported by you" (or similar) and exit gracefully.
6. **"No, different issue"** → proceed to create the new issue document normally, as described in Section 5.1.

> **Don't build a second voting system.** This is the single most common implementation mistake to avoid here: duplicate confirmation is NOT a new feature with its own counter. It reuses the exact same `votes` collection and the exact same 5-net-upvote threshold from Section 6.3. The only new logic is WHERE in the UI the vote gets triggered from (a pre-submission duplicate check screen, in addition to the issue detail page).

## 8. Map View

- Use Leaflet with OpenStreetMap tiles. Do not use the Google Maps JS API — it requires billing setup that isn't worth the time cost for this build.
- On report submission, capture `{ lat, lng }` via the browser Geolocation API as described in Section 5.1.
- Render every non-deleted issue as a pin on the map. Color/icon-code pins by status using the corrected label set from Section 3.2 and the table in Section 9.4 below.
- Clicking a pin opens that issue's detail view (Section 9.1).
- No proximity/"near me" filtering. No reverse geocoding (do not convert coordinates to a readable street address). The map simply pans and zooms freely, centered on the city by default, showing every pin. This was a deliberate scope cut to save build time — do not add either feature.
- Include a legend (as shown in Image 3) listing all 5 status pins plus the Disputed indicator, using distinct icon shapes per status (not color alone) — this is the accessibility requirement from Section 3.5, and Image 3's legend already does this well; keep that approach.

## 9. Issue Detail, Feed, and Dashboard

### 9.1 Issue detail page

- Show: photo, category, severity, AI-generated short description, current status badge, Disputed badge if applicable.
- Show a timeline component covering all 5 pipeline stages (not just the 3 shown in Image 4) with the corrected labels from Section 3 / 9.4.
- Show upvote/downvote controls and current net vote count. The meaning and threshold of this control changes depending on current stage — at `reported` it's working toward 5 net votes to reach `verified`; at `resolved_pending_verification` it's working toward 3 net votes (and any downvote sets `disputed`). Make sure the UI communicates which voting context is active, since the same visual component is reused for both.
- If status is `in_progress` and `citizenFixable` is true, OR if the logged-in user is `org_admin` (any category), show a "Submit resolution proof" action that uploads a photo and moves status to `resolved_pending_verification`.

### 9.2 Feed / issue list

- Scrollable list of issues, most recent first.
- Filterable by category (all 6, not the 4 shown in Image 5) and by status (all 5 states, not the 3 shown in Image 5).
- Each card: thumbnail photo, title/short description, status badge with corrected label, Disputed badge if applicable, relative timestamp.

### 9.3 Impact dashboard

- Stat cards: total issues reported (all-time), resolved count, pending count (sum of `verified` + `in_progress` + `resolved_pending_verification`).
- Bar chart: issue count by category, across all 6 categories.
- Bar or pie/donut chart: issue count by status, across all 5 real pipeline states (Reported, Verified, In Progress, Pending Verification, Resolved) — per the correction in Section 3.1, do NOT include a "Rejected" segment, and DO include "Verified."
- All figures computed from real data in the `issues` collection (after seeding per Section 6.4) via aggregation queries — not hardcoded placeholder numbers.

### 9.4 Final reconciled status label table (use this exact wording everywhere)

| Pipeline stage (internal/DB value) | Label to display in UI | Where it appears |
|---|---|---|
| `reported` | "Reported" | Map pin, issue badge, feed badge, dashboard chart |
| `verified` | "Verified" | Map pin, issue badge, feed badge, dashboard chart |
| `in_progress` | "In Progress" | Map pin, issue badge, feed badge, dashboard chart |
| `resolved_pending_verification` | "Pending Verification" (or "Resolved – Pending Verification" on the map specifically, to disambiguate from the earlier reported→verified pending state) | Map pin, issue badge, feed badge, dashboard chart |
| `resolved` | "Resolved" | Map pin, issue badge, feed badge, dashboard chart |
| `disputed: true` (boolean flag, NOT a pipeline stage) | "Disputed" badge, shown IN ADDITION to whatever stage it's attached to — never replaces a stage label | Issue detail page, feed card, map pin (small overlay indicator) |
